Status: fix03 is locally implemented and tested; **neither Unbound fix03 nor the
Solas fallback is visually accepted yet**. The grid under
`runs/evidence/iter04_lighting_e8ed51b_fix02` honestly proves the new snow scene,

failed.

## Deterministic runtime shader options

Bootstrap already sorted the configured options, but that was insufficient in two
independent ways:

1. `lookdev_render_batch.sh` runs profiles with `--no-bootstrap`. The fix02 noon
   instance manifest was still generated at `2026-07-10T11:46:32Z`, before commit
   48e3501 changed the night options. Its runtime sidecar proves stale values:
   `LIGHT_NIGHT_I=1.50`, `LIGHT_NIGHT_B=1.40`,
   `ATM_NIGHT_I=1.30`, and `LIGHTSHAFT_NIGHT_I=140`; all moon-phase controls added
   by 48e3501 were absent. The configured fix02 values were `2.00`, `1.80`, `1.80`,
   `200`, plus the moon-phase controls. Scene state is resolved at run time, which
   is why the same grid can contain the new snow overlay while still using the old
   shader sidecar.
2. Iris rewrites `shaderpacks/<exact shader ZIP>.txt` through Java
   `Properties.store` when the client exits. The six stale runtime mappings were
   mutually identical, but their complete SHA-256 values differed because the
   first lines were timestamps such as `#Fri Jul 10 16:01:56 UTC 2026`; removing
   only that line gave the same SHA-256 for all six payloads:
   `644bc8626d737a411a78f7eaef07358aeb426efcf95a945fb0fb14fdcf80d65a`.

fix03 closes both holes:

- immediately before every run, rewrite `iris.properties` and the selected shader
  sidecar from the **current resolved profile**, even when the instance already
  exists and the batch uses `--no-bootstrap`;
- after the client terminates, parse the actual Java-properties bytes, including
  continuations and Java escapes;
- preserve Iris' observed mapping (including legitimate default omission or runtime
  canonicalization), sort its keys, apply deterministic Java escaping, use LF
  newlines and omit all wall-clock comments;
- atomically write the fixed `# Generated by mcdata.` byte format and allow the
  operation to run safely more than once;
- freeze those exact observed bytes as
  `run_dir/shader_options.runtime.properties` and record their SHA-256 in
  `pipeline.jsonl`, so later runs cannot overwrite the evidence in a shared instance.

This preserves an Iris-readable properties file and cannot hide a runtime option
drift by substituting requested values after the fact: the normalized bytes are the
actual observed values. Unit tests cover an exact byte golden,
Unicode/control/punctuation escaping, the real timestamp/CRLF shape, idempotence,
observed-value preservation, duplicate-key rejection without mutation, and stale
pre-run replacement from the current profile. A new remote run must still prove
both that the selected night controls survived at their requested values and that
all six post-runtime files have one literal SHA-256; local tests are not runtime
acceptance.

change noon, golden hour, rain and snow too, so fix03 does not use that shortcut.

## fix03 bright, cool values

All six controlled Unbound profiles still inherit one shader-options mapping and
therefore one sidecar payload. Only night-specific values changed from fix02:

| Control | fix03 selected value |
|---|---:|
| `LIGHT_NIGHT_R/G/B/I` | `1.00 / 1.60 / 2.00 / 2.00` |
| `ATM_NIGHT_R/G/B/I` | `1.00 / 1.60 / 2.00 / 2.00` |
| `LIGHTSHAFT_NIGHT_I` | `200` |
| full / partial / dark moon | `2.00 / 1.60 / 1.20` |

Blue, the two intensity scalars, full-moon influence and night light shafts are at
their legal maxima. Red and green intentionally remain below `2.00` to retain a
cool moonlight ordering instead of producing neutral white. At the full-moon
endpoint the effective terrain multipliers rise from fix02's
`2.40 / 3.90 / 5.40` to `4.00 / 6.40 / 8.00` (R/G/B), before the shader's later
lighting operations. There is no `TM_EXPOSURE` override.

This is the strongest controlled Unbound attempt, but it cannot promise stronger
directional separation because the pack has no night-only control for that degree
of freedom. The next grid must independently judge brightness and visible cool
directional shaping on floors/facades; a brightness-only improvement is not enough
to mark the midnight endpoint accepted.

## Snow accumulation scene invariant

Both snow endpoints still declare the same `snow_accumulation_v1` scene variant and
the same ordered `scene.additions` payload. fix03 does not alter them:

- all showcase walking surfaces at absolute `y=63` are replaced in-place with
  `minecraft:snow_block`;
- both reflecting-basin water surfaces at absolute `y=63` are replaced in-place
  with `minecraft:ice`;
- all twelve mutations remain structured `fill ... replace ...` entries;
- no block is added above the existing surface, so route height and the canonical
  429-cell obstacle footprint remain unchanged;
- clear snow and snowfall differ only in `weather`; ordinary noon, golden-hour,
  midnight and rain still contain no snow additions.

The existing tests retain the exact 12-addition count, y-level, command ordering,
snow/ice block set, no-additions assertions for noon/rain, time-only noon/midnight
state and weather-only clear-snow/snowfall state.

## Snow particle limitation

Unbound 5.8.1 exposes weather opacity, general close-particle reduction, rain style
and improved-rain controls, but no snowflake-size or snow-only particle-scale
control. Changing shared opacity/general particle controls would also alter the
accepted rain behavior. Oversized vanilla snow sprites therefore remain a known
limitation; fix03 does not claim to repair them.
